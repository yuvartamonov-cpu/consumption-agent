# dayexp — Расходы за сегодня

> **Обновлено 2026-05-17 (День 1 спринта):** добавлен единый формат логов
> через `ScanMetrics` из `imap_folders.py` — на каждый IMAP-аккаунт
> логируется `folders_scanned`, `messages_seen`, `messages_deduped`,
> `messages_parsed`, `errors`, `elapsed_s`. Это упрощает диагностику
> «почему `/dayexp` не нашёл письмо».

## Использование

```bash
python3 <skill-dir>/scripts/dayexp_report.py        # только сегодня
python3 <skill-dir>/scripts/dayexp_report.py -n 7    # последние 7 дней
```

Скрипт:
1. Запускает `daily_cheque_scan.py` из `consumption_agent/` — сканирование всех почт + SMS
2. Читает из БД `consumption.db` все записи за N дней (включая сегодня)
3. Выводит отчёт в формате:
   - Заголовок: «📊 Расходы за сегодня» или «📊 Расходы за последние N дн.»
   - Количество покупок и общая сумма
   - Каждая покупка: иконка источника, магазин, сумма, описание
   - Блок «📌 По магазинам:» — итоги по каждому магазину

## Команда Telegram

Команда `/dayexp [N]` в `telegram_bot.py`:
- `/dayexp` — расходы за сегодня (по умолчанию N=1)
- `/dayexp 7` — расходы за последние 7 дней
- Отвечает «🔍 Сканирую почты и SMS за последние N дн....»
- Асинхронно запускает `daily_cheque_scan.py`
- После сканирования **редактирует** сообщение с результатом
- Если Telegram не принимает Markdown-разметку отчёта или вопроса о дубле, бот автоматически откатывается в plain text вместо зависания на сообщении «Сканирую...»
- Если Telegram API временно недоступен или отвечает `TimedOut`, edit/send должны идти через короткие bounded-retry, а не оставлять команду навсегда на сообщении «Сканирую...»
- Использует `asyncio.create_subprocess_exec` (не блокирует бота)

## Формат SMS в отчёте

- Для SMS не показывается сырой текст и баланс карты
- Показываются только магазин и сумма
- Время из SMS выводится отдельной последней строкой: `🕐 HH:MM`

## Зависимости

- `consumption_agent/consumption.db` — SQLite БД
- `consumption_agent/daily_cheque_scan.py` — скрипт сканирования почт и SMS
- `consumption_agent/imap_folders.py` — выбор релевантных IMAP-папок (`INBOX`, `Spam/Junk`, папки чеков)
- Таблица `purchases` с полями: `purchase_date`, `total_amount`, `store_name`, `source`, `notes`, `deleted_at`

## IMAP-охват

`daily_cheque_scan.py` сканирует не только `INBOX`, а все релевантные папки, найденные через `imap_folders.py`:
- `INBOX`
- `Spam` / `Junk` / `Спам`
- папки с чеками вроде `Receipts`, `Checks`, `чеки`

Дедупликация между папками идёт по `Message-ID`, чтобы одно письмо не учитывалось дважды.

## Дедупликация расходов

`/dayexp` использует общий модуль `purchase_dedup.py` для дедупликации между email и SMS:
- канонизирует алиасы магазина, например `Умный ритейл` → `Самокат`
- режет повтор, если совпали дата, канонический магазин и время операции
- учитывает отдельную `доставку` в email-чеке: одинаковыми считаются `итог` и `итог без доставки`, если SMS содержит только списание за товары
- не делает агрессивный дедуп без времени операции, чтобы не схлопывать разные покупки в один день
- если пользователь пометил дубль через кнопку удаления, такой soft-deleted расход больше не должен повторно импортироваться при следующем `/dayexp`
- для SMS+email с близкими суммами (например Самокат: чек с доставкой и SMS без доставки) бот задаёт вопрос о возможном дубле отдельным сообщением

## Единые правила поиска

Подробные правила строгой фильтрации и антидубля для всех 4 почт и SMS теперь описываются **только** в [email-access/SKILL.md](../email-access/SKILL.md), раздел `Строгие правила поиска и антидубль`.

`dayexp` должен следовать этим правилам:
- импортировать только письма с явными маркерами чека/списания
- не считать маркетинговые письма расходами, даже если в них есть сумма
- дедупить одинаковые SMS между двумя телефонами до импорта
- в спорном кейсе лучше пропустить слабый сигнал, чем создавать ложный дубль
