## SMS — Phone Link

SMS с двух телефонов читаются через SQLite-базу Microsoft Phone Link:

```
/mnt/c/Users/*/AppData/Local/Packages/Microsoft.YourPhone_8wekyb3d8bbwe/LocalCache/Indexed/*/System/Database/phone.db
```

Конвертация Windows FILETIME → datetime:
```python
def windows_ticks_to_datetime(value):
    unix_seconds = (int(value) - 116444736000000000) / 10_000_000
    return datetime.fromtimestamp(unix_seconds)
```

Базу надо копировать в temp (с -wal, -shm) — основной файл может быть заблокирован.

## Сканирование чеков — готовый скрипт

`daily_cheque_scan.py` — ежедневное сканирование всех 4 почт + SMS:

```bash
cd ~/.openclaw/workspace/consumption_agent
source venv/bin/activate
source .env
python3 daily_cheque_scan.py
```

Скрипт:
1. Подключается ко всем 4 почтам через IMAP
2. Обходит релевантные папки: `INBOX`, `Spam/Junk/Спам`, папки чеков вроде `Receipts` / `Checks` / `чеки`
3. Ищет письма за сегодня (+ вчера для первой синхронизации)
4. Определяет магазин по отправителю (Самокат, Ozon, WB, Яндекс и т.д.)
5. Парсит HTML-чеки Платформы ОФД (сумма, дата, товары)
6. Для писем без HTML-чека — ищет сумму регулярками
7. Сканирует SMS из Phone Link на предмет расходов
8. Добавляет записи в `purchases`
9. Дополнительно режет дубли между папками по `Message-ID`
10. Логирует всё в `logs/daily_cheque_scan.log`

## Дедупликация email + SMS

Для расходов используется общий helper `consumption_agent/purchase_dedup.py`:
- межпапочная дедупликация email по `Message-ID`
- межисточниковая дедупликация email/SMS по дате, каноническому магазину и времени операции
- нормализация алиасов продавца, например `УМНЫЙ РИТЕЙЛ` и `Самокат` считаются одним магазином
- если email-чек содержит отдельную строку `доставка`, то дедуп сравнивает и полный итог, и сумму без доставки
- без времени операции запись автоматически не схлопывается, чтобы не убить разные покупки одного дня

## Строгие правила поиска и антидубль

Этот раздел — **единый источник правил** для всех сканов почт и SMS. Другие skills должны ссылаться на него, а не копировать логику.

### 1. Что сканировать в почте

Для каждого из 4 ящиков сканировать только:
- `INBOX`
- `Spam` / `Junk` / `Спам`
- папки чеков: `Receipts`, `Checks`, `чеки` и похожие, найденные через `imap_folders.py`

Не сканировать вручную произвольные архивы, если задача не требует backfill.

### 2. Какие письма можно импортировать

Импорт разрешён только если письмо проходит оба условия:
- отправитель/subject матчится на известный магазин или сервис
- в письме есть **признак платежа/чека**, а не просто сумма

Допустимые маркеры платежа:
- `кассовый чек`
- `электронный чек`
- `receipt`
- `чек`
- `оплата`
- `оплачено`
- `платеж`
- `покупка`
- `списание`
- `списано`
- `итого`
- `к оплате`
- `сумма заказа`
- `order total`

Если этих маркеров нет, письмо не импортировать, даже если там есть сумма в рублях.

### 3. Какие письма блокировать как маркетинг

Если письмо содержит маркеры рассылки/акции и не содержит явного признака чека, его надо пропускать:
- `промокод`
- `скидка`
- `акция`
- `подборка`
- `новинки`
- `подарок`
- `кэшбэк`
- `баллы`
- `бонус`
- `выгода`
- `распродажа`
- `рекомендуем`

Примеры, которые нельзя импортировать в расходы:
- Ozon / WB / Яндекс.Маркет письма с промокодом и суммой скидки
- письма Яндекс Плюса вроде `Новинки мая уже на Кинопоиске`

### 4. Правила по Яндекс Плюсу

`Яндекс Плюс` импортировать только если есть явный сигнал списания/чека:
- `списан` / `списали` / `списание`
- `оплата` / `оплачено` / `платеж`
- `автопродление` / `продлен`
- `электронный чек` / `кассовый чек`

Письма про контент, подборки, фильмы, сериалы, новинки — не импортировать.

### 5. Правила извлечения суммы

Сумму из письма брать только из строк с контекстом оплаты:
- `итого`
- `сумма заказа`
- `сумма к оплате`
- `к оплате`
- `оплачено`
- `оплата`
- `списание`
- `платеж`
- `order total`
- `amount charged`

Нельзя брать "любое первое число с ₽" из письма. Это основной источник ложных расходов из промо-рассылок.

### 6. Правила по SMS

SMS-расход импортировать только если выполняется одно из условий:
- сообщение пришло от известного банковского/платёжного отправителя
- в тексте есть явный расходный маркер: `покупка`, `списание`, `оплата`, `дебет`
- удалось выделить сумму и уверенный магазин/категорию

Правила жёсткости:
- в `daily_cheque_scan.py` использовать **один** SMS-конвейер: `sms_expense_monitor.py`; не запускать параллельно legacy `scan_sms_today()`, иначе появятся дубли
- generic `SMS payment` без уверенного магазина не импортировать
- одинаковые SMS с двух телефонов дедупить по нормализованной паре `sender + body`
- потом дополнительно дедупить через БД по `дата + канонический магазин + время + сумма`

### 7. Порядок дедупликации

Применять в таком порядке:
1. `Message-ID` между папками и папка/ящик-дубликаты
2. `sender + normalized body` для одинаковых SMS с двух Phone Link БД
3. `date + canonical_store + event_time + amount`
4. для сервисов доставки дополнительно сравнивать `итог` и `итог без доставки`

### 8. При спорном кейсе

Если сигнал слабый:
- лучше **пропустить письмо/SMS**, чем создать ложный расход
- потенциальный дубль показывать отдельно как вопрос пользователю, а не импортировать второй записью молча

## При необходимости — ручной запуск

```python
# Подключение к одной почте:
import imaplib
from imap_folders import discover_target_mailboxes

imap = imaplib.IMAP4_SSL('imap.gmail.com', timeout=20)
imap.login('yu.v.artamonov@gmail.com', pwd)
folders = discover_target_mailboxes(imap)
for mailbox in folders:
    imap.select(f'"{mailbox}"', readonly=True)
    result, data = imap.search(None, '(ON 12-May-2026)')
```

## Примечания

- **VPN**: для доступа к IMAP с WSL должен быть включён VPN на Windows
- **Phone Link**: база доступна только из WSL (через /mnt/c/)
- **Дубли**: скрипт использует `purchase_dedup.py`, а не простую проверку по `(purchase_date, total_amount, store_name)`
- **SMS**: 59 900+ без названия магазина → отсеивается как перевод между счетами
