# План кодирования на 5 дней — consumption_agent

Дата: 16.05.2026
Основание: текущее состояние consumption_agent, seller-link search в Memory Lane, расширенный IMAP folder scan для /dayexp, /monthexp, /debts, /fines.

## Цель спринта

За 5 дней не распыляться на весь backlog, а довести до рабочего уровня два критических направления:
- качество и устойчивость пользовательских команд отчётов/долгов по почте;
- retrieval quality для поиска товаров из Memory Lane с упором на прямые ссылки продавцов и official/distributor entry points.

## День 1 — Наблюдаемость и тестируемость почтового сканирования

Что делаем:
- Добавляем единый лог-формат для daily_cheque_scan.py, credit_monitor.py, scripts/fines_bot.py.
- Логируем список выбранных IMAP-папок и метрики: folders_scanned, messages_seen, messages_deduped, messages_parsed.
- Пишем mock-based integration tests на IMAP LIST/SELECT/SEARCH/FETCH.

Файлы:
- daily_cheque_scan.py
- credit_monitor.py
- scripts/fines_bot.py
- imap_folders.py
- tests/test_imap_folders.py
- новые integration tests

Результат дня:
- есть доказуемый тестами проход по INBOX, Spam, Receipts;
- легче диагностировать, почему команда что-то не нашла.

## День 2 — Official / distributor retrieval для Memory Lane

Что делаем:
- Выделяем слой official/distributor resolver поверх текущего seller-link search.
- У брендов ищем не просто generic search URL, а более точные entry points: официальный сайт, российский дистрибьютор, авторизованный retailer.
- Улучшаем source ordering: official > distributor > strong retailer > generic marketplace link.

Файлы:
- ml_providers.py
- ml_search_v2.py
- опционально новый ml_official_sites.py
- tests для routing / provider generation

Результат дня:
- по брендовым вещам /ml_search начинает отдавать более точные точки входа, а не только поисковые ссылки.

## День 3 — Качество перевода запросов для AliExpress / Alibaba

Что делаем:
- Расширяем translation map для категорий одежды, дома, техники, косметики.
- Добавляем перевод признаков: material, fit, sleeve, season, gender, style, color variants.
- Нормализуем брендовые кавычки и убираем русские хвосты из foreign queries.

Файлы:
- ml_providers.py
- tests/test_ml_providers.py

Результат дня:
- английские search URLs становятся пригодными для реального foreign retrieval.

## День 4 — Vision fallback для чеков

Что делаем:
- Вводим fallback после OCR, когда текст явно шумный или confidence низкий.
- Отделяем fast path от expensive path: хорошие чеки остаются на Tesseract, плохие идут в Vision.
- Дожимаем парсинг item lines, total, delivery fee.

Файлы:
- telegram_bot.py
- receipt_handler / cheque_parser.py / связанные OCR-модули
- tests на проблемные чеки

Результат дня:
- меньше мусорных распознаваний, выше шанс получить структурированный чек из фото/скрина.

## День 5 — Завершённый пользовательский loop

Приоритетный вариант A:
- price-drop alerts для товаров из Memory Lane: watchlist, повторный поиск, уведомление в Telegram.

Резервный вариант B:
- reminders/replacement workflow для items: nearing replacement → alert → кнопка подтверждения.

Результат дня:
- у пользователя появляется один законченный сценарий с кнопкой, записью в БД и понятным Telegram UX.

## Что не брать в этот 5-дневный слот

- автоматизацию Ozon как обязательную часть поиска;
- большие рефакторы архитектуры без user-facing результата;
- параллельный старт нескольких незавершённых направлений из long-tail backlog.

## Критерии успеха по итогу 5 дней

- почтовые команды можно отлаживать по логам и тестам, а не только по симптомам;
- Memory Lane search заметно лучше на branded товарах и official/seller links;
- foreign marketplace queries не выглядят как полу-русская мешанина;
- чековый pipeline лучше справляется с трудными изображениями;
- хотя бы один новый пользовательский workflow завершён end-to-end.
