# Матрица Потребностей (Needs Matrix)

## Подход

Классическая мультипликативная формула (Frequency × Priority × Urgency) не работает для
реальных жизненных сценариев. Вместо неё — **байесовская модель**: условная вероятность
потребности в момент времени t.

```
P(need | season, calendar, inventory_state, budget_state, profile, market_state)
```

Это не сложнее линейной формулы в реализации (LLM + набор правил), но качественно честнее:
куртка получает высокий вес в сентябре, лекарство от аллергии — в сезон цветения, а подарок
на годовщину привязывается к календарю, а не к частоте.

## Оси системы

### Ось 1 — Природа потребности (Nature)

| Тип | Код | Пример |
|-----|-----|--------|
| Расходник | `consumable` | Еда, бытовая химия, бритвенные кассеты |
| Актив (долгий) | `asset` | Ноутбук, куртка, мебель |
| Услуга / подписка | `service` | Стриминг, страховка, облако |
| Импульс / впечатление | `impulse` | Увидел, понравилось, отложил |
| Цифровой актив | `digital` | Книги, лицензии ПО, игры |
| Недвижимость | `realty` | Жильё, парковка, кладовка |
| Событие | `event_driven` | Подарок, праздник, путешествие |

### Ось 2 — Категория (иерархическая)

Дерево категорий, неограниченная вложенность. Категория определяет набор типовых атрибутов
(шаблон для LLM/CV) и temporal patterns (см. Ось 5).

```
Все товары и услуги
├── Одежда и обувь
│   ├── Верхняя одежда (пуховик, пальто, куртка)
│   ├── Повседневная (футболки, джинсы, рубашки)
│   ├── Обувь
│   │   ├── Кроссовки / кеды
│   │   ├── Ботинки / сапоги
│   │   └── Туфли / лоферы
│   ├── Аксессуары (ремни, часы, сумки, рюкзаки)
│   └── Бельё и домашнее
├── Техника и электроника
│   ├── Компьютеры и планшеты
│   ├── Аудио и видео
│   ├── Телефоны и носимые
│   ├── Бытовые приборы
│   ├── Кухонная техника
│   └── Умный дом
├── Продукты питания
├── Косметика и уход
├── Здоровье и аптека
├── Дом и ремонт
├── Спорт и активный отдых
├── Авто и транспорт
├── Хобби и развлечения
├── Цифровое
└── Животные
```

### Ось 3 — Жизненный цикл (Lifecycle Stage)

```
IMPRESSION                ← "увидел, запомнил"
   │
   ▼
WISHLIST                  ← "хочу, изучаю"
   │
   ▼
PURCHASED                 ← куплено
   │
   ▼
INVENTORY → IN_USE         ← используется
   │          │
   │          ├── LOW_STOCK ← заканчивается (расходники)
   │          ├── EXPIRING  ← срок годности на исходе
   │          └── WARRANTY_EXP ← гарантия истекает
   │
   ├── REPLACED             ← заменено
   └── DISPOSED             ← утилизировано
```

### Ось 4 — Приоритет / бюджет (Priority)

Уровни те же: `critical (10)` → `must (7)` → `planned (5)` → `backlog (3)` → `wish (1)`.
Но приоритет — **один из входов в байесовскую модель**, а не множитель в формуле.

### Ось 5 — Temporal Pattern (вместо «Frequency»)

Вместо линейной шкалы «частота использования» — паттерны поведения во времени:

| Паттерн | Описание | Примеры |
|---------|----------|---------|
| `continuous` | Используется непрерывно, с равномерным расходом | Зубная паста, стиральный порошок, кофе |
| `seasonal` | Зависит от сезона / погоды | Куртка, солнцезащитный крем, шины |
| `sparse` | Редкие, непредсказуемые покупки | Инструмент для ремонта, чемодан |
| `event_driven` | Привязан к событиям (календарь, праздники) | Подарки, цветы, поездки |
| `lifecycle` | Привязан к сроку службы другой вещи | Фильтры для пылесоса, картриджи |
| `subscription` | Регулярные платежи | Стриминг, страховка |

Temporal pattern определяет, **когда агент должен оценивать потребность**, а не просто
«насколько часто пользуются».

### Ось 6 — Атрибуты (Attributes) — гибкая схема

Каждая категория имеет шаблон атрибутов (JSONB). Шаблон — это набор подсказок для LLM
при распознавании, а не жёсткая структура.

### Ось 7 — Связи (Dependencies)

| Тип связи | Описание | Пример |
|-----------|----------|--------|
| `requires` | А требует Б | Кофемашина → таблетки, фильтры |
| `enables` | А делает Б возможным | Принтер → картриджи, бумага |
| `consumes` | А расходует Б | Авто → топливо, масло, шины |
| `replaces` | А заменяет Б | Новый телефон → старый уходит в disposed |
| `complements` | А хорошо сочетается с Б | Пальто → шарф |
| `upgrades` | A улучшает Б | SSD → компьютер |
| `alternative` | А — альтернатива Б | Электрическая зубная щётка ↔ обычная |

**Граф потребностей** — ключевое отличие от простого каталога. Позволяет агенту:
1. При покупке кофемашины — предложить расходники.
2. При окончании фильтров — напомнить докупить.
3. При замене телефона — предложить рассмотреть продажу старого.
4. Отслеживать lifecycle: купил принтер → агент ждёт, когда понадобятся картриджи.

### Ось 8 — Канал обнаружения (Discovery Source)

Как потребность появилась. Важно для понимания собственного поведения.

| Канал | Описание |
|-------|----------|
| `email_parsed` | Из письма о доставке (MVP-канал) |
| `receipt_scan` | Скан чека |
| `manual` | Ввёл руками |
| `photo_impression` | Увидел и сфоткал |
| `voice_memo` | Продиктовал голосом |
| `link_shared` | Скинул ссылку |
| `screenshot` | Скриншот рекламы/поста |
| `dependency` | Выведено из связи с другой потребностью |
| `agent_proactive` | Агент сам предложил на основе анализа |

## Кластеры образа жизни (вместо гендерного разделения)

Ось «пол» заменена кластерами образа жизни. Влияют на начальные веса категорий,
temporal patterns и приоритеты.

| Кластер | Влияние |
|---------|---------|
| **Профессия** (офисная / удалёнка / физическая / творческая / разъездная) | Определяет типичную одежду, инструменты, характерные расходы |
| **Наличие детей** (нет / маленькие / школьники / подростки) | Определяет категории (игрушки, детская одежда, питание), частоту и бюджет |
| **Климат** (умеренный / холодный / жаркий / сезонный) | Определяет вес сезонных категорий |
| **Жильё** (квартира / дом / аренда) | Определяет категории (инструменты, мебель, ремонт) |
| **Транспорт** (авто / общественный / велосипед / пешком) | Определяет связанное потребление |
| **Хобби / активность** (спорт / готовка / сад / игры / искусство) | Определяет вес соответствующих категорий |
| **Ценности потребления** (minimalist / quality-focus / value-focus / trend-following) | Определяет подход к выбору и бюджет |

Кластеры не хардкодятся — они собираются из данных о покупках (ML-кластеризация профиля)
и могут пересматриваться со временем. На старте — простой опросник при инициализации.
