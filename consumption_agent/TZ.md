---

## 1. Назначение

Персональная система инвентаризации и отслеживания жизненного цикла покупок. Собирает данные с почты (чеки Ozon), ведёт каталогизированный инвентарь с категориями, сроками годности, гарантиями и уведомлениями.

**Цель:** не терять информацию о купленных вещах — когда куплено, сколько стоит, где лежит, когда закончится гарантия/срок годности, пора ли перезаказать.

## 3. Модель данных

### 3.1. profiles

Профили пользователей (один дефолтный, расширяемо).

- `id` — TEXT PRIMARY KEY ('default')
- `name`, `currency`, `timezone`, `notification_config`

### 3.2. categories

Двухуровневое дерево категорий (34 шт).

- Родительские: одежда, техника, продукты, косметика, здоровье, дом, спорт, авто, хобби, цифровое, животные, подписки
- Дочерние: верхняя одежда, обувь, компьютеры, аудио, телефон, бытовая техника, кухонная техника, корм, ветеринария, зоотовары, мебель, декор, кухня, спорттовары, книги, интимные, прочее, лекарства, витамины

### 3.3. purchases

Покупка (чек / заказ). **Всего: 54 покупки (из них 35 связаны с товарами).**

- `purchase_date`, `total_amount`, `currency`, `store_name`, `order_number`
- `source` — 'ozon' | 'ozon_pdf' | 'ozon_pdf_cheque' | 'local'
- `email_message_id` — UNIQUE, для дедупликации
- `receipt_url` — ссылка на чек
- `deleted_at` — soft delete (все активны)

### 3.4. items

Конкретный товар. **Всего: 494 активных.**

- `name`, `brand`, `model`, `sku`
- `category_id` → categories (все проставлены)
- `status` — 'in_use' (все)
- `quantity`, `unit`, `remaining`
- `purchase_date`, `purchase_price`, `purchase_id` → purchases (35 шт)
- `warranty_months` (20 шт), `expiry_date`, `lifespan_months` (12 шт)
- `tags` — JSON-массив
- `data_origin` — 'local' (основной), 'cheque_parse' (из чеков)
- `deleted_at` — soft delete

### 3.5. alerts

Уведомления. **Всего: 0.** (ни одна гарантия/срок не истекли и не истекают)

- `alert_type` — warranty_expiring / warranty_expired / expiry_approaching / expired / low_stock / price_drop / seasonal_reminder / dependency_alert / budget_warning
- `status` — pending / sent / dismissed / actioned

### 3.6. Вспомогательные таблицы

- `cheques_log` — 83 записи (лог импортированных чеков, дедупликация по email_uid)
- `recognized_items_log` — 2428 записей (лог распознанных товаров из OCR, не сопоставлены)
- `subscriptions` — не заполнена

### 3.7. Индексы (7 шт, созданы)

- `idx_items_deleted`, `idx_items_category`, `idx_items_purchase`
- `idx_items_warranty` (условный), `idx_items_lifespan` (условный)

## 5. Потоки данных

### 5.1. Импорт чеков (import)

```
Gmail INBOX → IMAP search (FROM sender.ozon.ru) →
  фильтр по теме («Ваш чек») →
  парсинг Subject/Date → запись в purchases + cheques_log
```

- **Макс. писем:** 60 за запуск
- **Дедупликация:** по email_message_id (UNIQUE)
- **receipt_url:** извлекается из HTML письма (ссылка /e-check/download/)
- **source:** 'ozon_pdf' если есть ссылка на чек, иначе 'ozon'
- **Проблема:** query-параметры URL срезаются через `.split('?')[0]` (может сломать ссылку)

### 5.2. Парсинг фискальных чеков (parse)

```
cheques_log (source='ozon_pdf') → скачивание receipt_url →
  извлечение: дата, сумма, товары (название × количество × цена)
```

- **Формат:** кассовый чек с ИТОГ и строками `№. Название Qty × Цена ≡ Сумма`
- **Результат:** новые покупки (data_origin='ozon_pdf_cheque') + товары в БД
- **Скачивание:** urllib.request.urlretrieve → файл receipt_{id}.txt
- **Match с существующими:** по LIKE `%{name[:30]}%`
- **Не тестирован** на реальных PDF-чеках Ozon (формат может отличаться)

### 5.3. Обогащение (enrich)

```
items → категоризация по ключевым словам из словаря UNC (33 ключа) →
  гарантии (техника/спорт: 6-24 мес) →
  сроки годности (продукты/здоровье/косметика: 12-36 мес) →
  алерты (только pending, старые не трогаем)
```

- **Категория по умолчанию:** cat_other («Прочее»)
- **Алерты:** создаются только если срок/гарантия истекли или истекают (30 / 90 дней)

### 5.4. Матчинг (match)

```
recognized_items_log → нормализация названий →
  exact-поиск по items (если rapidfuzz нет) →
  fuzzy-поиск ≥85% (если rapidfuzz есть) →
  запись matched_item_id + метаданные
```

- **2428 записей** в логе, ни одна не сопоставлена
- **Без rapidfuzz** — работает только точное совпадение (скорее всего 0 результатов)
- Мусорные записи отфильтровываются
