# Архитектура персонального Consumption Manager

## Общая концепция

Персональный агент управления потреблением — система, которая ведёт полный lifecycle всех
материальных и нематериальных активов владельца: от момента, когда вещь привлекла внимание,
до утилизации. Агент действует на стороне владельца, в его инфраструктуре (OpenClaw).

## Принципы

1. **Zero-friction input** — владелец не заполняет формы. В идеале просто живёт, а агент собирает данные сам (email, чеки, покупки на маркетплейсах).
2. **Privacy-by-design** — разграничение локальных / облачных / агрегированных данных на уровне схемы, retention policy, consent per event.
3. **Extensible taxonomy** — новая категория товаров не требует кода. Достаточно записи в дереве категорий + набора атрибутов.
4. **Proactive, not annoying** — агент предлагает, а не требует. Частота и формат уведомлений настраиваются.
5. **MVP-first** — сначала узкий полезный slice, потом расширение. Никакой 8-осевой матрицы до проверки гипотезы.
6. **MCP/A2A-ready** — архитектура закладывает совместимость с emerging протоколами (Model Context Protocol, Agent-to-Agent), но не требует их для MVP.

## Поток данных

```
ВХОДНЫЕ КАНАЛЫ
  │
  ├── 📧 Email (письма о доставках с маркетплейсов) — приоритетный канал MVP
  ├── 🧾 Сканы чеков / фото чеков
  ├── 🔗 Ссылки на карточки товаров
  ├── 💬 Чат (Telegram) — ручной ввод
  ├── 📸 Фото товаров (распознавание)
  └── 🎤 Голосовые заметки (ASR → текст)
      │
      ▼
ОБРАБОТКА
  │
  ├── Быстрый цикл (fast path):
  │   ├── Категоризация (LLM / embedding)
  │   ├── Извлечение ключевых атрибутов (бренд, модель, цена, дата)
  │   └── Сохранение в сыром виде
  │
  └── Отложенный энричмент (lazy, по триггеру):
      ├── Глубокий парсинг характеристик
      ├── Поиск аналогов на маркетплейсах
      ├── Мониторинг цен
      └── Матчинг с существующим инвентарём
      │
      ▼
ХРАНЕНИЕ (Privacy tiers)
  │
  ├── [LOCAL] — на устройстве владельца
  │   ├── Inventory — что есть (актуальное имущество)
  │   ├── Purchases Journal — история покупок
  │   └── Profile — предпочтения, ограничения, бюджеты
  │
  ├── [CLOUD_OPT_IN] — зашифровано, только с согласия
  │   ├── Wishlist — что хочется (согласие на облачное хранение)
  │   ├── Impressions — лента впечатлений
  │   └── Price history — трекинг цен
  │
  └── [ANONYMOUS_AGGREGATE] — обезличенная статистика
      (опционально, только с явного согласия)
      │
      ▼
ЛОГИКА (Agent Layer)
  │
  ├── 🧮 Needs Engine — байесовская оценка потребности: P(need | season, calendar, inventory, budget, profile)
  ├── ⏰ Lifecycle Tracker — отслеживание гарантий, сроков годности, износа
  ├── 💵 Budget Controller — контроль расходов, лимиты по категориям
  ├── 🔗 Dependency Engine — связи между потребностями (кофе → кофемашина → фильтры)
  └── 💡 Recommendation Engine — проактивные предложения
      │
      ▼
ВЫВОД
  │
  ├── 💬 Чат (Telegram) — основной канал общения
  ├── 🔔 Push-уведомления — гарантии, скидки, напоминания
  └── 📊 Дашборд — статистика, инвентаризация
```

## MCP / A2A-совместимость

Архитектура закладывает интеграционный слой на базе emerging протоколов:

- **MCP (Model Context Protocol)** — для подключения внешних AI-инструментов (поиск, анализ цен, CV)
- **A2A (Agent-to-Agent)** — для взаимодействия с другими агентами (семья, агрегированный спрос)
- **ACP (Agent Communication Protocol)** — для обмена данными между инстансами

В MVP интеграции не требуется, но слой абстракции (Gateway Adapter) закладывается сразу, чтобы не переписывать интеграции позже.

## Двухстадийная обработка впечатлений (Memory Lane)

```
Входящее впечатление (фото, голос, скриншот, ссылка)
  │
  ├── [Fast Path] — обязателен, выполняется сразу
  │   ├── Категоризация: "это одежда / техника / еда / …"
  │   ├── Embedding: векторное представление для будущего поиска
  │   ├── Raw-описание: LLM извлекает что увидел в 1-2 предложениях
  │   └── Сохранение: impressions.recognition_status = 'raw'
  │
  └── [Lazy Enrichment] — только по триггеру
      ├── Триггеры:
      │   • Владелец явно спросил: "найди мне это"
      │   • Предмет помечен как высокоприоритетный
      │   • Алгоритм обнаружил совпадение (по embedding) с active wishlist
      │   • Истекает время жизни записи → напоминание
      ├── Полное распознавание: бренд, модель, цена
      ├── Поиск аналогов на маркетплейсах
      └── Сохранение: impressions.recognition_status = 'enriched'
```

~60–70% впечатлений никогда не превратятся в покупку. Двухстадийная модель экономит ресурсы и не нагружает систему бесполезным энричментом.
